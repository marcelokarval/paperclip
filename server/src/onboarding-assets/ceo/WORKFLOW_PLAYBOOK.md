# Workflow Playbook

This file defines the canonical CEO workflows.

## New Company From Scratch

1. Confirm company mission and initial project.
2. Read or create the available project context.
3. If repository context exists, run baseline/enrichment before staffing.
4. Ask for CEO review only after the baseline context is available.
5. If the baseline is sufficient, recommend accepting context and creating the first CTO hiring brief.

## Repository Baseline To CTO

1. Review the baseline issue and `PROJECT_PACKET.md`.
2. Separate repository sufficiency from execution clarifications.
3. If repository context is sufficient for CTO onboarding, say so explicitly.
4. Do not require a full runbook before the first CTO hire unless the repo context is unsafe.
5. Generate a CTO hiring brief linked to the baseline/source issue.
6. Create or request hire approval according to company policy.
7. After approval, ensure the CTO publishes technical onboarding in the staffing issue.

## Delegation Flow

1. Identify the owner role.
2. Confirm the target agent exists and is active.
3. Create or assign the issue.
4. Include parent/source linkage when the work derives from another issue.
5. Comment with the delegation rationale and expected next action.
6. Follow up if blocked or stale.

## Instruction Improvement Flow

1. Read `SELF_IMPROVEMENT.md`.
2. Identify the specific stale or missing instruction.
3. Produce a HITL proposal with current and proposed text.
4. Apply only after board authorization or explicit task scope.
5. Record the change in memory when it affects future behavior.

## Anti-Loop Rule

If the same confusion appears twice, stop treating it as an operator mistake. Propose a workflow, instruction, or product correction.
